# 第6章：使用 GenAI 开发软件应用

## 一、传统 ML 开发 vs Prompt-Based 开发

### 传统机器学习开发的痛点

Andrew Ng 以一个非常直观的方式对比了两种开发范式。在传统监督学习中，构建一个 AI 功能需要经历漫长的过程：

1. **收集标注数据**——通常需要数月甚至更久
2. **训练模型**——需要 ML 工程师调参、选模型、处理过拟合等
3. **评估与迭代**——每次改动都需要重新训练
4. **部署**——需要模型服务化、API 封装

一个简单的文本分类项目，从立项到上线，**数月是常态**。而最大的瓶颈往往是**标注数据**——你需要成千上万条人工标注数据，成本高、周期长、迭代慢。

### Prompt-Based 开发的革命性变化

GenAI 时代，这一模式被彻底颠覆。对于许多 NLP 任务，你不需要收集任何标注数据，也不需要训练模型——你只需要写一个 prompt。

> **从"写代码训练模型"到"写提示词告诉模型你要什么"**

项目的开发周期从**数月缩短到数天甚至数小时**。这不是渐进式改进，而是范式转换。

## 二、实战对比：情感分析系统

Andrew 在课程中用一个真实的对比案例来说明这种差异。

### 传统方法：Supervised Learning

- 需要标注 10,000+ 条评论为 positive/negative
- 训练一个分类器（如 BERT fine-tuning）
- 开发周期：2-3 个月
- 迭代成本高——每改一次需求，可能需要重新标注数据

### Prompt-Based 方法

- 写一条 prompt：`"Classify the sentiment of the following review as positive or negative. Review: {text}."`
- 不需要任何训练数据
- 开发周期：**几小时**
- 迭代成本极低——改几行 prompt 即可

### 对比不是完美的

当然这不是说传统 ML 没有用了。Andrew 也坦诚地指出了 prompt-based 方法的局限性：

- **大规模部署的成本**：LLM 推理成本远高于传统小模型
- **延迟**：LLM 调用通常是百毫秒到秒级，传统模型可以做到微秒级
- **可解释性与可控性**：传统模型的行为更可预测

所以正确的选择策略是：**原型阶段用 prompt 快速验证，生产阶段根据需求选择最佳方案**——可能是 LLM、可能是微调模型、也可能是传统小模型。

## 三、GenAI 应用的生命周期

Andrew 提出了一个清晰的开发流程框架：

### Step 1: 明确任务（Define the Task）

这是最重要的一步。你需要清晰定义：

- 模型要完成什么任务？（分类？生成？摘要？翻译？）
- 输出格式是什么？（JSON？自然语言？标签？）
- 质量标准是什么？（准确率？相关性？风格一致性？）

### Step 2: 设计 Prompt（Design the Prompt）

根据任务编写 prompt，包括：

- 系统指令（System message）：告诉模型角色和约束
- 用户指令（User message）：具体任务描述
- 示例（Few-shot）：可选，帮助模型理解格式
- 输入数据：要处理的实际内容

### Step 3: 评估（Evaluate）

评估是容易被忽略但极其关键的一步。Andrew 强调**不要只在 2-3 个例子上觉得"看起来不错"就上线**。系统评估包括：

- **开发集 evaluation**：准备 50-100 个测试样本，系统性地评估输出质量
- **自动化评估**：可以用 LLM-as-a-judge（让另一个 LLM 评估输出质量）
- **人工评估**：关键场景需要人工审核

### Step 4: 迭代（Iterate）

根据评估结果改进：

- **修改 prompt**：最常见的优化方式
- **调整参数**：temperature、top_p、max_tokens 等
- **增加上下文**：few-shot 示例、RAG 等
- **换模型**：有时换一个更强的模型是最高效的优化

### Step 5: 部署（Deploy）

上线后在真实场景中持续监控。注意：**开发时的评估分布和上线后的真实分布可能完全不同**，需要持续追踪。

## 四、评估指标：不止是准确率

传统 ML 的评估指标比较明确（如准确率、F1-score），但 GenAI 应用需要更立体的评估体系：

### 1. 准确性（Accuracy）

模型的输出是否正确。对于分类任务比较直接，对于生成任务需要对"正确"做更细致的定义。

### 2. 相关性（Relevance）

输出是否与用户的输入相关。一个常见问题是 LLM 会"跑题"——回答了问题但答非所问。通过 ROUGE、BLEU 等指标或者 LLM-as-a-judge 来评估。

### 3. 安全性（Safety）

Andrew 特别强调了这一点。GenAI 应用可能产生：

- **有害内容**：仇恨言论、暴力、不当内容
- **幻觉（Hallucination）**：编造事实
- **隐私泄露**：意外暴露训练数据中的敏感信息

安全评估需要专门的 red-teaming（红队测试）和安全护栏（guardrails）。

### 4. 鲁棒性（Robustness）

小幅修改输入是否会大幅改变输出？一个好的 prompt 应该对输入噪声有一定的容忍度。

---

## 五、本章小结

1. **Prompt-based development** 将 AI 应用开发周期从数月缩短到数天，这是一个范式级别的变革。
2. 传统 ML 和 prompt-based 方法各有优劣，最佳实践是根据场景灵活选择——**原型用 prompt，生产按需决策**。
3. 清晰的**开发流程**（定义→设计→评估→迭代→部署）是构建可靠 GenAI 应用的基础。
4. 评估 GenAI 应用需要多维度的指标，安全性应当作为**首要考量**而非事后补充。
