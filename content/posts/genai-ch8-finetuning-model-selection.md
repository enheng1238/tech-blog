# 第8章：微调、预训练与模型选择

## 一、预训练（Pre-training）——从零开始的庞然大物

### 什么是预训练？

Pre-training 是从头训练一个大语言模型的过程。数据集通常是**整个互联网的文本**——万亿级别的 tokens。训练成本极高，动辄数千万到数亿美元。

Andrew Ng 用了一个很好的类比：预训练就像是**让一个孩子读遍全世界的书**，从而获得对语言和知识的广泛理解。

### 为什么要提到预训练？

大多数人不需要做预训练。Andrew 提到它的目的是让你理解 LLM 的能力来源：

- 预训练给了模型**语言能力**和**世界知识**
- 这个过程极其昂贵（GPU 集群运行数月、电力、冷却、硬件折旧）
- 只有像 OpenAI、Google、Meta、Anthropic 这样的公司才有资源做

> 对于 99% 的开发者/企业来说，**你不需要预训练自己的模型**。你只需要在现有模型的基础上做微调或 prompt engineering。

---

## 二、微调（Fine-tuning）——站在巨人肩上

### 什么是 Fine-tuning？

Fine-tuning 是在**已经预训练好的模型**基础上，用特定领域的数据进一步训练，让模型更好地完成特定任务。

### Fine-tuning 能做什么？

1. **改变输出风格/格式**：比如让模型总是输出 JSON 格式、特定语气
2. **学习特定领域知识**：虽然 RAG 更适合"引入知识"，但 fine-tuning 可以让模型"更擅长"某个领域
3. **优化特定任务表现**：如代码生成、翻译、分类任务
4. **减少 prompt 长度**：经过 fine-tuning 的模型不需要那么长的 prompt

### Fine-tuning 的误区

Andrew 特别澄清了一个常见误解：**Fine-tuning 不是用来给模型注入新知识的**。如果你的目标仅仅是让模型知道 2024 年发生的事情，应该用 RAG 而不是 fine-tuning。为什么？

- Fine-tuning 可能会产生**知识冲突**（新数据与原始训练数据矛盾）
- 模型在 fine-tuning 中**记忆效果不可控**——它可能学不会，也可能过拟合
- RAG 的方式更可控、可溯源、可更新

Fine-tuning 擅长的是**行为改变**（behavior change），而不是**知识注入**（knowledge injection）。

## 三、Instruction Tuning 与 RLHF

### Instruction Tuning（指令微调）

基础预训练模型（base model）只知道"预测下一个 token"，并不懂得"遵循指令"。Instruction Tuning 就是用大量的（指令，期望输出）对来训练模型学会**按指令行事**。

### RLHF（Reinforcement Learning from Human Feedback）

RLHF 是让模型更进一步的关键技术，ChatGPT 背后的核心技术之一。流程：

1. **收集人类偏好数据**：对同一 prompt，让人类标注哪条输出更好
2. **训练 Reward Model**：训练一个模型来预测人类会偏好哪个输出
3. **用强化学习微调**：PPO 算法，让 base LLM 学会生成 Reward Model 评分高的回答

RLHF 让模型学会了**什么是有帮助的（helpful）、真实的（honest）、无害的（harmless）**回答——这就是 AI alignment（AI 对齐）的核心。

## 四、如何选择模型——多维度决策框架

Andrew Ng 给出了一个非常实用的决策框架，不是单纯地说"选这个"或"选那个"，而是从多个维度综合考量：

### 1. 性能（Performance）

更强的模型（如 GPT-4、Claude 3.5）在复杂推理、代码生成、多语言等方面表现更好。但对于简单任务，小模型可能已经足够。

> **规则：先用最强的模型验证可行性，再用最小的模型满足需求。**

### 2. 成本（Cost）

- **推理成本**：每 1K tokens 的价格。小模型（如 GPT-4o-mini、Claude Haiku）价格只有旗舰版的 1/10 到 1/50。
- **训练成本**：Fine-tuning 的成本取决于数据量和模型大小。
- **基础设施成本**：自部署 vs API 调用。

### 3. 延迟（Latency）

实时应用（如聊天机器人）对延迟敏感。小模型推理更快。API 调用还有网络延迟。

### 4. 数据安全（Data Security / Privacy）

这是很多企业最关心的维度：

- **API 调用**：数据会发送到第三方服务器
- **私有部署（On-Premise）**：数据不出公司网络，但需要自建基础设施
- **数据不被用于训练**：部分 API 提供商承诺不将你的数据用于训练

### 5. 可控性（Control）

你能多大程度上定制模型的行为？开源模型可以 fine-tune、修改架构、替换 tokenizer；闭源 API 只能通过 prompt 控制。

## 五、开源 vs 闭源模型

### 闭源模型（Closed-Source）

- **代表**：GPT-4、Claude、Gemini
- **优点**：开箱即用、性能最强、无需基础设施
- **缺点**：数据离开公司、vendor lock-in、成本随规模线性增长

### 开源模型（Open-Source）

- **代表**：Llama（Meta）、Mistral、Qwen、Falcon
- **优点**：数据安全、可定制、无 vendor lock-in、可本地部署
- **缺点**：需要技术团队维护、初期投入高、性能通常弱于顶配闭源模型

### Andrew 的建议

不要有"开源崇拜"或"闭源偏好"。决策应该基于：

- **你的数据有多敏感？**
- **你的团队有多强的 ML 工程能力？**
- **你的预算/成本结构是怎样的？**
- **你需要多强的模型能力？**

对于初创公司和小团队，先走 API 路线（快速验证），规模大了之后再考虑自部署。

## 六、Tool Use 与 Agents

### 让 LLM 调用外部工具

LLM 本身是一个"文本到文本"的引擎。但通过 **tool use**，它可以调用外部工具来突破自身限制：

- **计算器**：做精确计算（LLM 不擅长数学）
- **代码解释器**：执行代码并获取结果
- **API 调用**：获取实时数据、发邮件、操作数据库
- **搜索引擎**：获取最新信息

实现方式：在 prompt 中定义一组函数/工具的接口规范，LLM 根据用户需求选择调用哪个工具以及传入什么参数。

### Agents——自主行动的 AI

当 tool use + LLM 推理能力 + 记忆（memory）组合在一起，就构成了所谓的 **Agent**——一个可以**自主规划、执行、反思**的 AI 系统。

Andrew 认为 Agents 是 GenAI 的下一个重要方向：不是简单的"你问我答"，而是模型可以**主动采取行动**来完成复杂任务。

---

## 七、本章小结

1. **预训练**成本极高，**99% 的人不需要做**——站在已有模型的肩膀上即可。
2. **Fine-tuning** 改变的是模型的**行为**（风格、格式、任务适应性），而不是知识库（那应该用 RAG）。
3. **RLHF** 是让模型变得有用、真实、无害的关键技术，也是 ChatGPT 成功的核心。
4. 模型选择是一个**多维度决策**：性能、成本、延迟、安全、可控性缺一不可。
5. **Tool use 和 Agents** 正在开启 GenAI 的下一个阶段——从聊天到自主行动。
