# 第9章：工具使用与 AI Agent——从对话到行动

## 引言：LLM 的边界

大语言模型（LLM）擅长对话和生成文本，但它在很多基础能力上存在天然局限：无法进行精确数学运算、无法访问实时信息、无法操作外部系统。第9章的核心问题正是：**如何突破 LLM 自身的边界，让它从「会说话」进化到「能办事」？**

答案是 **Tool Use（工具使用）** 与 **AI Agent（智能体）**。

---

## Tool Use：给 LLM 配上工具箱

Tool Use 的核心思路很简单：不让 LLM 自己去执行它不擅长的任务，而是让它学会「调用外部工具」。Andrew Ng 用几个经典场景做了说明：

| 能力短板 | 工具方案 | 示例 |
|---------|---------|------|
| 数学计算 | 调用计算器 API | `calculate("3.14 * 2^10")` |
| 实时信息 | 调用搜索 API | `web_search("今天 NVIDIA 股价")` |
| 代码执行 | 调用 Python 解释器 | `run_python("data.describe()")` |
| 图像生成 | 调用图像模型 API | `generate_image("赛博朋克城市")` |

**实现机制**：LLM 的输出中嵌入特殊格式的函数调用标记（如 JSON 格式的 `{"function": "search", "params": {"q": "..."}}`），由外围系统解析并执行，将结果返回给 LLM 继续推理。

这种方法的关键洞察是：**LLM 不需要成为全能模型，它只需要成为一个聪明的「路由器」**——判断什么时候该用哪个工具，然后把结果整合进自己的回答中。

---

## 从 Tool Use 到 AI Agent

单个工具调用是有用的，但真正的能力跃升发生在 **Agent** 层面。Agent 是能够自主完成以下循环的系统：

1. **感知（Perceive）**：接收用户指令和环境反馈
2. **规划（Plan）**：将复杂任务分解为子步骤
3. **执行（Act）**：调用工具完成每个子步骤
4. **观察（Observe）**：获取工具执行结果
5. **循环（Loop）**：根据结果决定下一步行动

这其实就是 **ReAct 模式（Reason + Act）**——一种让 LLM 交替进行推理和行动的框架。

### ReAct 模式详解

ReAct 的全称是 **Reasoning and Acting**。传统的做法是让 LLM 一次输出完整答案（Chain-of-Thought），但 ReAct 的做法是：

```
思考（Thought）：用户要分析这份销售数据，我需要先读取文件。
行动（Action）：调用 read_file("sales.csv")
观察（Observation）：返回了 1000 行销售记录...
思考（Thought）：数据量较大，先做汇总统计。
行动（Action）：调用 run_python("df.describe()")
观察（Observation）：平均销售额 3520 元，标准差...
思考（Thought）：好的，我可以生成报告了。
最终回答：...
```

这个循环的好处是：**每一步的「思考」都是可解释的**，系统在哪个环节出错可以被追溯和修正。

## 应用场景

### 自动数据分析 Agent
用户上传 CSV，Agent 自动探索数据、做可视化、生成报告。整个过程不需要用户编写一行代码。

### 代码 Agent（Coding Agent）
像 GitHub Copilot、Claude Code、Cline 这类工具，本质上是 Agent：理解需求 → 搜索代码库 → 编写代码 → 运行测试 → 发现错误 → 修改代码 → 直到通过。

### 客服 Agent
处理退款、查询订单、修改地址等操作，需要调用 CRM 系统、支付系统、物流系统等多个后端 API。Agent 可以按顺序或并行调用这些工具。

---

## AI Agent vs 传统软件

Andrew Ng 强调了一个重要区别：

| 传统软件 | AI Agent |
|---------|---------|
| 固定流程（if-else 决策树） | 动态决策（LLM 自主判断） |
| 输入格式严格 | 自然语言输入 |
| 容错性低，异常需要 hardcode | 可自我纠正和重试 |
| 维护成本随复杂度指数增长 | 维护成本增长较平缓 |

但这不意味着 Agent 优于所有传统方案。**在确定性强、安全性要求高的场景（如飞行控制系统），传统软件的确定性反而是优势。**

---

## 实际注意事项

1. **Token 消耗**：ReAct 循环会消耗大量 token，每次循环都需要将之前的上下文重新传入
2. **错误传播**：一个工具的错误输出会影响后续所有决策，需要有健壮的错误处理
3. **安全边界**：给予 Agent 的工具权限需要严格限制，防止恶意使用
4. **超时机制**：防止 Agent 陷入无限循环，需要设置最大步骤数

---

## 小结

第9章的核心洞见是：**LLM 的真正价值不来自它「知道」多少，而来自它能「调用」什么。** Tool Use 将 LLM 从信息孤岛变成了系统的中枢，Agent 则让这个中枢具备了自主规划和行动的能力。这是从「聊天机器人」到「AI 同事」的关键一步。
