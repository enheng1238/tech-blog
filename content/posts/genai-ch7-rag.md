# 第7章：RAG — 检索增强生成

## 一、RAG 是什么？

RAG 全称 **Retrieval-Augmented Generation（检索增强生成）**，是 Andrew Ng 在课程中重点讲解的一种核心架构模式。它的思路非常直观：

> **当模型不知道答案时，不要让它硬猜——先检索相关信息，再把信息注入 prompt，让模型基于这些信息回答。**

RAG = 检索（Retrieval）+ 生成（Generation），两阶段的流水线：

1. **Retrieval 阶段**：从一个外部知识库中检索与用户问题最相关的文档/片段
2. **Generation 阶段**：将检索到的信息与原始问题一起注入 LLM，生成最终回答

之所以说这个架构重要，是因为它解决了 LLM 最根本的局限性之一：**模型的知识固化在其训练数据中**。

---

## 二、RAG 解决的两个核心问题

### 问题一：训练数据截止

所有 LLM 都有知识截止日期。比如一个模型训练于 2023 年，它不知道 2024 年发生了什么。如果你问它"2024 年奥运会的金牌榜"，它要么答不上来，要么就**幻觉（hallucination）**——编造一个看似合理但实际错误的答案。

RAG 通过**实时检索最新信息**，让模型可以回答训练数据截止之后的问题，相当于给模型装上了"联网能力"。

### 问题二：无法访问企业私有数据

企业的内部文档、数据库、CRM 系统——这些数据 LLM 在训练时完全看不到。而 RAG 允许企业：

- 将自己的文档（PDF、Wiki、Confluence、Notion 等）存入向量数据库
- 当员工提问时，先检索内部知识库
- 检索结果 + 问题 → LLM → 基于企业内部知识的回答

这意味着**你不需要重新训练或微调模型**，就能让 LLM 掌握企业的私有知识。

---

## 三、RAG 的工作流程

Andrew 在课程中详细拆解了 RAG 从用户提问到生成回答的完整流程：

### Step 1: 用户提问

用户输入一个问题，比如："我们公司的年假政策是什么？"

### Step 2: 检索相关知识库

系统将用户的问题转化为向量（通过 Embedding 模型），然后在**向量数据库**中搜索最相似的文档片段。这里的关键是：

- **Embedding（嵌入）**：将文本转化为一个高维向量，语义相近的文本在向量空间中距离更近
- **向量数据库**（Vector Database）：专门存储和检索向量的数据库，如 Pinecone、Weaviate、ChromaDB、FAISS 等
- **相似度搜索（Similarity Search）**：找到与问题语义最接近的 Top-K 个文档片段

### Step 3: 注入 Prompt

将检索到的文档片段作为上下文（context）注入到 prompt 中。典型的 prompt 模板如下：

```
Use the following context to answer the user's question.
If the context doesn't contain the answer, say you don't know.

Context:
{retrieved_documents}

Question: {user_question}
Answer:
```

请注意指令中的 **"If the context doesn't contain the answer, say you don't know"**——这是减少幻觉的关键设计。

### Step 4: LLM 生成回答

LLM 基于注入的上下文和用户的问题，生成最终的答案。因为答案来源于检索到的实际文档，**输出有据可查、可溯源、大大降低幻觉风险**。

## 四、RAG vs Fine-Tuning

这是 Andrew 重点澄清的一个对比——很多初学者会把两者混为一谈。

| 维度 | RAG | Fine-Tuning |
|------|-----|-------------|
| **目标** | 引入新知识 | 改变模型行为/格式/风格 |
| **数据需求** | 不需要标注数据，只需要文档 | 需要高质量的标注数据集 |
| **知识更新** | 实时——更新向量库即可 | 需要重新训练 |
| **成本** | 低成本（只需检索 + 推理） | 高成本（训练 + 部署） |
| **幻觉控制** | 强——输出基于检索内容 | 弱——模型可能遗忘或编造 |
| **适用场景** | 知识密集型（问答、客服、文档查询） | 格式/风格/行为转型（代码生成、翻译、特定输出格式） |

Andrew 给出的建议是：**如果目标是让模型"知道更多"——用 RAG。如果目标是让模型"行为不同"——用 Fine-Tuning。两者可以组合使用。**

## 五、向量数据库与 Embedding 的基本概念

### Embedding 是什么？

Embedding 是将文本（或图像、音频）转换为一个**固定长度的数值向量**。核心性质：**语义相近的文本，其向量在空间中距离更近**。

例如：
- "猫" 和 "狗" 的向量距离很近（都是宠物）
- "猫" 和 "操作系统" 的向量距离很远

### Embedding 模型

将文本转化为向量的模型称为 Embedding 模型，如 OpenAI 的 `text-embedding-3-small`、开源的 `BGE` 系列、`GTE` 系列等。

### 向量数据库

当文档数量很大（如上万份）时，不可能实时计算每个文档与问题的相似度。向量数据库通过高效的索引结构（如 HNSW、IVF）实现**毫秒级**的近似最近邻搜索（ANN）。

常见向量数据库：
- **Pinecone**：托管服务，上手快
- **Weaviate**：支持混合搜索（向量 + 关键词）
- **ChromaDB**：轻量级，适合本地开发
- **FAISS**：Meta 开源库，底层引擎

## 六、本章小结

1. **RAG = 检索 + 生成**，是解决 LLM 知识局限性的最实用方案。
2. 它有效解决了**训练数据截止**和**企业私有数据不可访问**两大痛点。
3. 核心流程：**提问 → 检索 → 注入 → 生成**，每一步都有优化空间。
4. RAG 和 Fine-Tuning 不是二选一的关系，而应该**根据需求组合使用**。
5. **Embedding + 向量数据库**是 RAG 的技术基石，理解它们的工作机制对搭建系统至关重要。
