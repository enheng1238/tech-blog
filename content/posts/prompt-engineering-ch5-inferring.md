# 第5章：推断（Inferring）

> 吴恩达 & OpenAI — ChatGPT Prompt Engineering for Developers

## 概述

本章重点展示 LLM 在**情感分析**与**实体识别**方面的强大推断能力。核心思想是：**用一个 prompt 完成多个推断任务**，无需为每个任务编写独立的分类器或查询语句。

---

## 5.1 什么是推断

推断（Inferring）是指 LLM 根据输入文本**推理出未明确写出的信息**，例如：

- 情感倾向（正面 / 负面）
- 情感类型（愤怒、喜悦、悲伤……）
- 实体提取（人名、产品名、公司名……）
- 主题分类
- 信息摘要式提取

传统做法需要为每个任务训练独立的模型或编写规则；而 LLM 可以在**同一个 prompt 中**同时完成多个推断，极大降低开发成本。

---

## 5.2 情感分析（Sentiment Analysis）

### 基本情感判断

将一段文本输入 LLM，询问其情感是正面还是负面：

```
Prompt:
对以下评论进行情感分析，输出 "正面" 或 "负面"：
"这个产品太棒了，我非常喜欢！"
```

### 情感维度细化

可以输出更细粒度的分析，例如同时提取：

- 情感极性（正面 / 负面）
- 情感强度（1–5 分）
- 具体情感类别（愤怒、悲伤、满足、兴奋……）

### 示例

> **输入文本：**  
> "客服等了 40 分钟才接通，态度还很差，完全没有解决问题。"
>
> **Prompt 输出示例：**
> - 情感：负面
> - 强度：4 / 5
> - 类别：愤怒、失望

---

## 5.3 实体识别（Entity Extraction）

LLM 可以从文本中提取特定类型的实体（人名、组织、地点、产品名等）。

### 示例

> **文本：**  
> "亚马逊的 Echo 智能音箱由 Jeff Bezos 领导下的团队开发，总部位于西雅图。"
>
> **Prompt 输出：**
> - 公司：亚马逊
> - 产品：Echo 智能音箱
> - 人物：Jeff Bezos
> - 地点：西雅图

---

## 5.4 一个 Prompt 做多个推断（Multiple Inferences in One Prompt）

本章**最核心的技巧**：将多个推断任务合并到一个 prompt 中，LLM 会一次性输出所有结果。

### 格式建议

有两种常见方式：

1. **结构化输出（推荐）**：指定 JSON、YAML 或固定分隔符格式
2. **分点列举**：用数字编号列出不同推断任务

### 示例 Prompt（三合一）

```
从以下评论中提取信息：
1. 情感极性（正面或负面）
2. 情感类别列表（如"愤怒""喜悦"）
3. 提到的所有产品名

评论：...
```

### 输出示例（JSON 格式）

```json
{
  "sentiment": "负面",
  "emotion_categories": ["愤怒", "失望"],
  "products": ["智能音箱 Echo"]
}
```

### 优势

- **节省 Token**：同一段上下文只需输入一次
- **降低延迟**：一次调用替代多次
- **一致性**：所有推断基于同一段理解，避免上下文漂移
- **开发效率**：无需训练多个分类器

---

## 5.5 主题分类（Topic Classification）

推断任务也可以是对整段文本进行主题分类。

### 示例

> **文本：**  
> "新款 MacBook 搭载了 M3 芯片，性能比上一代提升了 40%。"
>
> **推断输出：**  
> 主题：科技 / 硬件 / Apple

可以结合多标签分类，让 LLM 同时输出多个相关主题。

---

## 5.6 信息提取（Information Extraction）

与实体识别类似但更灵活——提取任何用户关心的字段。

### 示例

```json
{
  "product": "Echo Dot",
  "company": "Amazon",
  "price": "$49.99",
  "rating": 4.5,
  "review_count": 1234
}
```

提取格式**完全由 prompt 控制**，LLM 会按指定 schema 输出。

---

## 5.7 实战技巧与注意事项

### 1. 格式固化

指定输出格式时**提供示例**比单纯描述更可靠。

### 2. 处理不确定性

如果输入文本缺少某些信息，可以在 prompt 中指定默认值或 `null`：

```
如果未提及产品名，则输出 "未知"
```

### 3. 输出校验

对于生产环境，建议对 LLM 输出做简单的后处理校验（如 JSON 格式验证、字段存在性检查）。

### 4. Token 预算

结构化输出（尤其是 JSON）会占用较多输出 Token，在成本敏感场景下可考虑压缩成固定分隔符格式。

## 关键 Takeaways

| 概念 | 要点 |
|------|------|
| **多任务推断** | 一个 prompt 做情感分析 + 实体提取 + 主题分类 |
| **结构化输出** | 使用 JSON/YAML 格式让结果可编程消费 |
| **情感分析** | 极性、强度、类别均可一次获取 |
| **实体识别** | 人名、公司、产品、地点等 |
| **开发效率** | 替代传统多模型 / 多规则方案 |

> **核心洞见：** LLM 的推断能力使开发者能够用自然语言替代传统的分类 pipeline，将多步推断合并为一步，大幅简化数据清洗与分析流程。
