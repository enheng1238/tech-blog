## 2. 两大核心原则（复习）

### 原则一：编写清晰、具体的指令

> **模型不会读心术。你写得越模糊，输出就越随机。**

| 策略 | 说明 | 对应章节 |
|------|------|---------|
| **使用分隔符** | 用 `"""`、`---`、XML 标签等区分指令与输入 | 第2章 |
| **请求结构化输出** | 指定 JSON、HTML、Markdown 等格式 | 第2章 |
| **条件检查** | 让模型先检查条件是否满足，再执行任务 | 第2章 |
| **Few-Shot 示范** | 给出 1~3 个输入/输出示例引导模型行为 | 第2章 |

#### 示例对比

| 方式 | 提示词 | 效果 |
|------|--------|------|
| ❌ 模糊 | `总结这段文本。` | 输出格式不确定，可能过长或过短 |
| ✅ 清晰 | `用一句话总结以下文本，不超过50字："""..."""` | 输出精准可控 |

---

### 原则二：给模型时间「思考」

> **让模型逐步推理，而不是仓促给出结论。就像你考试时先打草稿再写答案一样。**

| 策略 | 说明 | 对应章节 |
|------|------|---------|
| **指定任务步骤** | 分解复杂任务，让模型按 1→2→3 的顺序执行 | 第2章 |
| **先思考后回答** | 让模型先推导出自己的答案，再与给定结论对比 | 第2章 |
| **Chain-of-Thought** | 要求模型「逐步思考」以提升复杂推理的准确性 | 第2章 |

#### 经典用例

```text
❌ 糟糕的 prompt：
判断以下结论是否正确：4 + 7 × 2 = 18

✅ 优秀的 prompt：
先计算出 4 + 7 × 2 的结果，然后将结果与 18 对比，最后判断结论是否正确。
列出你的计算步骤。
```

> **核心原理：** 模型在逐个 token 生成时，如果被要求「先推理再回答」，它的中间 token 会形成一个推理链，大幅降低最终的逻辑错误。

## 4. 课程核心洞见

### 4.1 Prompt Engineering 的本质

**提示工程不是「哄」模型，而是「编程」模型。**

- 提示词就是代码——清晰的结构、明确的边界、可复用的模式
- 迭代是开发的常态——没有一次写好的 Prompt，只有不断改进的 Prompt
- 输出格式就是 API 契约——用 JSON 等结构化格式让输出可被程序消费

### 4.2 LLM 的应用范式

课程展示了 LLM 的**四种典型应用模式**：

```
                LLM 应用模式
                    │
      ┌─────────────┼─────────────┐
      │             │             │
      ▼             ▼             ▼
  输入 → 摘要    输入 → 推断   输入 → 转换   输入 → 扩展
  （长变短）    （提取洞察）  （改变形式）  （短变长）
```

这四种模式覆盖了绝大多数实际应用场景：从客服总结（摘要）、评论分析（推断）、数据处理（转换）到内容生成（扩展）。

### 4.3 安全性提示

课程也多次强调了安全相关的提示技巧：

- **Prompt Injection 防护**：用分隔符隔离指令和外部输入
- **拒绝不当请求**：RLHF 对齐后的模型已被训练拒绝有害指令
- **边界条件处理**：当输入不满足条件时，让模型输出默认值或拒绝执行

## 5. 后续学习方向

Andrew Ng 在课程结尾给出了几个深入学习的方向：

### 5.1 更高级的提示技巧

| 技术 | 说明 | 学习资源 |
|------|------|---------|
| **Chain-of-Thought (CoT)** | 让模型逐步推理，提升复杂任务能力 | Wei et al. 2022 |
| **Tree-of-Thought (ToT)** | 多路径探索 + 回溯，解决更复杂的问题 | Yao et al. 2023 |
| **ReAct** | 推理 + 行动结合，让模型调用外部工具 | Yao et al. 2022 |
| **Automatic Prompt Engineering** | 用 LLM 自动优化 Prompt | Zhou et al. 2022 |

### 5.2 构建端到端应用

- **RAG（检索增强生成）**：将外部知识库与 LLM 结合
- **Agent / Function Calling**：让 LLM 调用 API、执行代码、操作数据库
- **多模态提示**：结合图像、音频等多模态输入

### 5.3 更深层的理解

- **模型内部机制**：Transformer 架构、Attention 机制、RLHF 原理
- **评估与对齐**：如何系统评估模型输出质量，确保安全性
- **成本优化**：Token 管理、缓存策略、模型蒸馏

### 5.4 推荐资源

| 资源 | 类型 | 适合人群 |
|------|------|---------|
| **DeepLearning.AI 其他课程** | 课程 | 想深入某个方向 |
| **OpenAI Cookbook** | 代码示例 | 想直接看代码最佳实践 |
| **Anthropic Cookbook** | 代码示例 | Claude 用户 |
| **LangChain / LlamaIndex 文档** | 框架文档 | 构建复杂 LLM 应用 |
| **PromptingGuide.ai** | 综合指南 | 系统学习提示技术 |

## 6. 笔记总结

### 6.1 一句话概括课程

> **Prompt Engineering = 理解模型 × 清晰指令 × 迭代优化**

### 6.2 最重要的两个习惯

| 习惯 | 做法 |
|------|------|
| **1. 写之前先想「模型知道什么」** | Instruction Tuned LLM 知道很多，但它不知道你期望的输出结构 |
| **2. 写完之后先问「还能不能更好」** | 迭代是 Prompt 开发的常态，不是异常 |

### 6.3 我的 5 个 Takeaways

1. **清晰 > 聪明** — 一个简单、具体的 Prompt 永远好过一个复杂、模糊的 Prompt
2. **不是你写得不好，是迭代次数不够** — 不要因为一次结果不好就否定 Prompt Engineering
3. **结构化输出改变一切** — 让 LLM 的输出变成可编程的 JSON，是打通 AI 与后端的关键
4. **系统角色是最强的杠杆** — 一个精心设计的 System Prompt 能持续引导整轮对话不走偏
5. **Prompt 工程师 = 新世纪的「程序员」** — 用自然语言「编程」，是 AI 时代最有价值的技能之一
